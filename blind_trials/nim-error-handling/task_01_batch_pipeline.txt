The goal is to judge error-boundary placement, retry behavior, and cleanup behavior.
The runtime behavior is fixed. Most API naming and internal decomposition decisions are yours.

## Provided fake environment

Include these local fake helpers exactly or with equivalent behavior:

```nim
type
  FakeReply = object
    transportError: string
    statusCode: int
    body: string

proc fakeLoadInput(id: string): string =
  if id.len == 0:
    raise newException(IOError, "input id empty")
  case id
  of "missing":
    raise newException(IOError, "input missing")
  of "blank":
    result = ""
  else:
    result = "doc:" & id

proc fakeSend(payload: string; attempt: int): FakeReply =
  case payload
  of "doc:retry-ok":
    if attempt == 1:
      FakeReply(transportError: "", statusCode: 429, body: "")
    else:
      FakeReply(transportError: "", statusCode: 200, body: "value=21")
  of "doc:transport-retry-ok":
    if attempt == 1:
      FakeReply(transportError: "timeout", statusCode: 0, body: "")
    else:
      FakeReply(transportError: "", statusCode: 200, body: "value=34")
  of "doc:parse-bad":
    FakeReply(transportError: "", statusCode: 200, body: "not-an-int")
  of "doc:http-bad":
    FakeReply(transportError: "", statusCode: 400, body: "")
  else:
    FakeReply(transportError: "", statusCode: 200, body: "value=" & $payload.len)

proc fakeAuditWrite(auditPath: string; line: string) =
  if auditPath == "audit-fail":
    raise newException(OSError, "audit write failed")

var closeCount = 0
var abortCount = 0

type
  FakeClient = ref object

proc newFakeClient(): FakeClient =
  FakeClient()

proc closeClient(client: FakeClient) =
  inc closeCount

proc abortClient(client: FakeClient) =
  inc abortCount

proc resetClientCounters*() =
  closeCount = 0
  abortCount = 0

proc clientCounters*(): tuple[closed: int, aborted: int] =
  (closeCount, abortCount)
```

## Required behavior

Your retry-limit parse helper must:

- write a parsed positive value into a `var` parameter on success
- return `true` on valid input
- return `false` on invalid input

Your retry predicate must:

- return `true` only when the current failure is retryable and the attempt is still below the retry limit
- retry transport errors `"timeout"` and `"disconnect"`
- retry HTTP status `429`, `408`, and `504`
- not retry other transport errors or HTTP statuses

Your final-failure classifier must map:

- `"timeout"` transport error to the exported timeout variant
- any other non-empty transport error to the exported network-error variant
- HTTP `429` to the exported rate-limit variant
- HTTP `408` or `504` to the exported timeout variant
- other non-success HTTP failures to the exported HTTP-error variant

Your batch runner must:

- process item ids in input order
- load input with `fakeLoadInput`
- treat empty loaded payload as an input failure without calling `fakeSend`
- otherwise call `fakeSend`
- retry while the retry predicate says to retry
- on HTTP `200`, parse only bodies of the form `value=<int>`
- treat parse failure as the exported parse-error variant
- record final non-success transport or HTTP responses using the final-failure classifier
- return one ordered outcome per input item unless audit logging fails fatally

Outcome rules:

- success outcome:
  - `success = true`
  - `value > 0`
  - no-error variant
  - empty error message
- failure outcome:
  - `success = false`
  - `value = 0`
  - non-no-error variant
  - non-empty error message

Attempt counting rules:

- attempts equal the number of calls made to `fakeSend` for that item
- input failures that never call `fakeSend` must still record `attempts = 1`

Audit rules:

- every failed item triggers `fakeAuditWrite(auditPath, line)`
- if audit logging raises `OSError`, catch it, add local context, and re-raise as `IOError`
- if audit logging fails, stop the batch immediately and let that exception escape

Your app runner must:

- create a `FakeClient`
- call the batch runner
- return the exported all-ok constant if every item succeeded
- return the exported partial-failure constant if the batch completed with at least one failed item
- return the exported fatal-runtime constant if a fatal exception escaped
- call `closeClient` exactly once on non-fatal completion
- call `abortClient` exactly once on fatal completion

## Required smoke run

Add a `when isMainModule:` block that uses your chosen exported API and performs all of these checks:

1. Retry-limit parse helper:
   - one valid parse
   - one invalid parse

2. Retry predicate:
   - HTTP `429` retries on attempt `1` with limit `2`
   - HTTP `429` does not retry on attempt `2` with limit `2`
   - transport error `"timeout"` retries on attempt `1` with limit `2`
   - HTTP `400` does not retry

3. Final-failure classifier:
   - `"timeout"` maps to the timeout variant
   - `"disconnect"` maps to the network-error variant
   - HTTP `429` maps to the rate-limit variant
   - HTTP `400` maps to the HTTP-error variant

4. Batch run on these ids with retry limit `2` and audit path `"audit.log"`:

```nim

  "good",
  "retry-ok",
  "transport-retry-ok",
  "parse-bad",
  "missing",
  "blank",
  "http-bad"
]
```

The smoke run must assert:

- ok count `3`
- fail count `4`
- seven outcomes in original input order
- `"retry-ok"` succeeds with `attempts == 2`
- `"transport-retry-ok"` succeeds with `attempts == 2`
- `"parse-bad"` ends in the parse-error variant
- `"missing"` ends in the input-error variant
- `"blank"` ends in the input-error variant
- `"http-bad"` ends in the HTTP-error variant

5. App cleanup:
   - after `resetClientCounters()`, running the app on `@["good", "retry-ok"]` returns the exported all-ok constant and leaves counters `closed = 1`, `aborted = 0`
