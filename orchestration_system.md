# Orchestration System: Nim Skill Benchmark Pipeline

This document is a fully actionable orchestration prompt. When given to an LLM agent with access to `sessions_spawn`, `exec`, `read`, `write`, and `edit` tools, it will execute a complete benchmark comparing two Nim skill files by generating code, compiling it, validating memory safety, and judging results.

## Configuration

Set these variables before starting. Adjust paths as needed.

```
REPO_ROOT = /tmp/skills_experiment
SKILL_ORIGINAL = trial_original_skill.md   # Human-written skill
SKILL_VERIFIED = trial_verified_skill.md   # AI-verified skill
NUM_TRIALS = 3                             # Trials per skill version
NIM_MM = orc                               # Memory model (--mm:orc)
TASK_SPEC = See TASK_SPEC section below
```

## Directory Layout

All artifacts go under `$REPO_ROOT/artifacts/`:

```
artifacts/
  original_1/  original_2/  original_3/
    subject_solution.nim      # Generated by Generator
    stress_test.nim           # Generated by Generator
    subject_solution          # Compiled binary (by Executor)
    stress_test_binary        # Compiled binary (by Executor)
    compile.log               # Compiler output
    valgrind.log              # Valgrind output
    asan.log                  # AddressSanitizer output
    run_output.log            # Stdout/stderr from test runs
    judge_verdict.json        # Final verdict from Judge
  verified_1/  verified_2/  verified_3/
    (same structure)
```

## TASK_SPEC

The exact task given to each Generator subagent:

```
Create a Nim source file called subject_solution.nim implementing a custom String type
with Reference Counting and Copy-on-Write (CoW) semantics using raw pointers and
manual memory management.

Required types (must match exactly):
  StrPayload* = object
    cap*, counter*: int
    data*: UncheckedArray[char]
  String* = object
    len*: int
    p*: ptr StrPayload

Required exports:
  proc initString*(s: var String; data: string)
  proc getStr*(s: String): string
  proc mutateAt*(s: var String; i: int; c: char) -- must trigger CoW when refcount > 1
  proc `=destroy`*(x: String) -- dec counter, dealloc when 0. Do NOT set fields to nil.
  proc `=wasMoved`*(x: var String) -- set p to nil, len to 0
  proc `=dup`*(b: String): String {.nodestroy.} -- share pointer, inc counter
  proc `=copy`*(a: var String; b: String) -- self-assign guard, destroy+wasMoved, share+inc

Must compile with: nim c --mm:orc. Export all. Only write subject_solution.nim.
```

## Phase 1: Generator Subagents

Spawn one Generator subagent per trial (6 total: 3 per skill version).

**For original skill trials (no skill injection):**

Spawn with `sessions_spawn`, `mode: "run"`, `cwd: "$REPO_ROOT/artifacts/original_N"`, and this task:

> You are a Nim programmer. Write subject_solution.nim in the current directory.
>
> [Insert TASK_SPEC here]
>
> Additionally, write stress_test.nim that imports subject_solution and tests these edge cases:
> 1. Repeated allocation and deallocation (loop 1000x: init, copy, mutate, destroy)
> 2. Self-aliasing: create a String, assign it to itself via =copy, verify data intact
> 3. Mutation-after-move: create String, move it, attempt mutation on moved-from source (should be no-op)
> 4. Deep copy chain: a → b → c, mutate c, verify a and b unchanged
> 5. Counter overflow protection: create 100 copies, destroy 99, verify counter is 1
> 6. Empty string handling: initString with "", getStr returns ""
>
> stress_test.nim must compile with `nim c --mm:orc --path:.` and print "ALL STRESS TESTS PASSED" on success.
> Write both files. Do not run them.

**For verified skill trials (with skill injection):**

Same task as above, but prepend this block before the TASK_SPEC:

> SKILL REFERENCE — follow these rules strictly:
>
> Ownership model: This is a shared/refcounted handle. =copy and =dup share the payload (increment counter), they do NOT deep-copy.
>
> Hook rules:
> - =destroy takes x: T (NOT var T). Decrement counter, dealloc when 0. Do NOT set fields to nil inside =destroy — that is =wasMoved's job.
> - =wasMoved takes x: var T. Set x.p = nil and x.len = 0.
> - =dup returns T with {.nodestroy.}. Share pointer, increment counter.
> - =copy takes (a: var T; b: T). MUST have self-assignment protection: if a.p == b.p: return. Then destroy(a), wasMoved(a), share b.p, inc counter.
> - Do NOT write a custom =sink. The compiler synthesizes it.
> - Never set fields to nil inside =destroy.
> - Self-assignment protection in =copy is mandatory.
> - {.error: "msg".} on =copy is ignored by the compiler. Use bare {.error.}.
>
After all Generator subagents complete, proceed to Phase 2.

## Phase 2: Executor Subagent

Spawn a single Executor subagent with `mode: "run"`, `cwd: "$REPO_ROOT"`:

> You are a build engineer. For each directory under artifacts/ (original_1 through verified_3):
>
> 1. Compile subject_solution.nim:
>    nim c --mm:orc --path:artifacts/DIR -o:artifacts/DIR/subject_solution artifacts/DIR/subject_solution.nim
>    Save all output to artifacts/DIR/compile.log
>
> 2. If compile succeeded, compile stress_test.nim:
>    nim c --mm:orc --path:artifacts/DIR -d:useMalloc -o:artifacts/DIR/stress_test_binary artifacts/DIR/stress_test.nim
>    Append output to artifacts/DIR/compile.log
>
> 3. For each compiled binary, run it and save output:
>    artifacts/DIR/subject_solution > artifacts/DIR/run_output.log 2>&1; echo "EXIT_CODE=$?" >> artifacts/DIR/run_output.log
>    artifacts/DIR/stress_test_binary >> artifacts/DIR/run_output.log 2>&1; echo "EXIT_CODE=$?" >> artifacts/DIR/run_output.log
>
> Report: for each trial, state whether subject_solution.nim compiled, whether stress_test.nim compiled, and what exit codes each binary produced.

## Phase 3: Validator Subagent

Spawn a single Validator subagent with `mode: "run"`, `cwd: "$REPO_ROOT"`:

> You are a memory safety validator. For each directory under artifacts/ (original_1 through verified_3):
>
> 1. Check if valgrind is available: which valgrind
> 2. If available, run each compiled binary under valgrind:
>    valgrind --leak-check=full --error-exitcode=42 artifacts/DIR/stress_test_binary > artifacts/DIR/valgrind.log 2>&1
>    Capture the exit code.
> 3. If valgrind is NOT available, compile with AddressSanitizer instead:
>    nim c --mm:orc --path:artifacts/DIR -d:useMalloc --passC:"-fsanitize=address -g" --passL:"-fsanitize=address" -o:artifacts/DIR/stress_asan artifacts/DIR/stress_test.nim
>    Run: artifacts/DIR/stress_asan > artifacts/DIR/asan.log 2>&1; echo "EXIT=$?" >> artifacts/DIR/asan.log
>
> For each trial, report:
> - Whether memory diagnostics passed (no leaks, no errors)
> - Any error output from valgrind or ASan
> - The exit code

## Phase 4: Judge Subagent

Spawn a single Judge subagent with `mode: "run"`, `cwd: "$REPO_ROOT"`:

> You are a strict judge evaluating Nim ownership hook implementations.
> For each directory under artifacts/ (original_1 through verified_3):
>
> Read these files if they exist:
> - artifacts/DIR/subject_solution.nim (the source code)
> - artifacts/DIR/compile.log (compilation results)
> - artifacts/DIR/run_output.log (runtime output)
> - artifacts/DIR/valgrind.log or artifacts/DIR/asan.log (memory diagnostics)
>
> Evaluate against these criteria. Score PASS or FAIL for each:
>
> 1. COMPILE: Does subject_solution.nim compile with --mm:orc? (FAIL if compile errors)
> 2. HOOK_SIGS: Does =destroy take x: T (not var T)? Does =wasMoved take var T?
> 3. NO_NIL_IN_DESTROY: Is there any field assignment (x.p = nil, x.data = nil) inside =destroy? (FAIL if yes)
> 4. SELF_ASSIGN: Does =copy have self-assignment protection? (FAIL if missing)
> 5. NODUP: Does =dup use {.nodestroy.}? (FAIL if missing)
> 6. COW: Does mutateAt implement CoW detach when counter > 1? (FAIL if missing)
> 7. STRESS_PASS: Did stress_test.nim pass all edge case tests? (FAIL if any test failed)
> 8. MEMORY_SAFE: Did valgrind/ASan report zero errors and zero leaks? (FAIL if any memory errors)
>
> Write verdict to artifacts/DIR/judge_verdict.json in this exact format:
> {
>   "trial": "original_1",
>   "skill_version": "original",
>   "criteria": {
>     "COMPILE": "PASS|FAIL",
>     "HOOK_SIGS": "PASS|FAIL",
>     "NO_NIL_IN_DESTROY": "PASS|FAIL",
>     "SELF_ASSIGN": "PASS|FAIL",
>     "NODUP": "PASS|FAIL",
>     "COW": "PASS|FAIL",
>     "STRESS_PASS": "PASS|FAIL",
>     "MEMORY_SAFE": "PASS|FAIL|SKIP"
>   },
>   "total_pass": 7,
>   "total_criteria": 8,
>   "notes": "..."
> }
>
> After evaluating all 6 trials, print a summary table comparing original vs verified skill results.

## Phase 5: Report Generation

After the Judge completes, read all `judge_verdict.json` files and write `benchmarking_results.md` with:

1. Summary table (trials × criteria)
2. Aggregate scores per skill version
3. Analysis of differences
4. Specific code quality observations

## Execution Constraints

- Each Generator subagent runs independently (can be spawned in parallel)
- Executor, Validator, and Judge run sequentially
- All file paths must be absolute
- Clean up workspace `subject_solution.nim` files after each Generator completes (they may write to workspace)
- If a subagent fails, mark that trial as FAILED and continue
- Never pass the gold standard implementation to any subagent

## Skill File Paths

- Original skill: `$REPO_ROOT/trial_original_skill.md`
- Verified skill: `$REPO_ROOT/trial_verified_skill.md`
